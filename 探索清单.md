## 三、六次失败实验的归因（探索报告核心素材）

### 失败 1：自一致性投票 N=3（DEV 81.1% → 81.1%，token ×3）

**预期**：单次 ~30% query 三次答案不一致，多数票能修一半 → +3-5%。
**实测**（DashScope）：分数完全不变。`prompt × 3.00, compl × 3.00` 严格等比 → 三次回答 token 数完全相同。
**初步归因**（事后修正）：DashScope qwen3-8b 同 prompt 输出 deterministic（疑似服务端 cache 或低温采样）。

**归因修正（基于 SiliconFlow 跨部署实验）**：通过切到 SiliconFlow 部署（已验证创意写作任务真采样，3 次输出完全不同）后，跑 539 条 dev N=3 投票（[output/20260507_220311/](../output/20260507_220311/)）：

- N=1 baseline @ SF: **79.4%**
- N=3 投票 @ SF: **79.8%（+0.4%，仅 ~2 条变化）**

→ 真采样部署上**投票收益也微乎其微**，**真正的根因不是部署 deterministic，而是 Qwen3-8B 在分类任务上 logits 极度 peaked**（首选 token softmax 概率 ≈ 1.0，即使真采样也几乎 always 选同一个）。这是**模型本身的特性**，评测方部署也不会消解。

**教训**：

1. 投票需要"决策边缘抖动"前提，但 Qwen3-8B 在结构化分类上极度自信（无论选对还是选错），所以投票从源头无效。
2. 跨部署对比意外暴露另一个事实：**同模型 DashScope 81.1% vs SiliconFlow 79.4%（-1.7%）**——不同 fp 精度 / kernel / sampling 实现导致 ~1-2% 浮动。**评测方分数不可从本地外推**。

### 失败 2：reasoning + label 二段输出（81.1% → 79.0%）

**预期**：让模型先 short reasoning 再出 label，强迫"对照 example 与 LABEL_SET 思考"，对近义 label 选错有效 → +3-5%。
**实测**：-2.1%。
**归因**：Qwen3-8B 是小模型，**reasoning 反作用**——它直觉对的题被错误推理带偏（"This text mentions 'charge'..."→ 选 `extra_charge_on_statement` 而非正确的 `direct_debit_payment_not_recognised`）。reasoning 收益的临界规模通常在 30B+。
**教训**：CoT 不是普适增强；小模型上验证后再用。

### 失败 3：per-label cap=2 → 1 + few-shot 倒序（81.1% → 79.0%）

**预期**：cap=1 让 8 条 example 覆盖 8 个不同 label（多样性 ↑）；倒序让最相关 example 紧贴 query（recency bias）→ +1-3%。
**实测**：-2.1%。
**归因**：每 label **双 example 是 sweet spot**——"两条同 label 的 example"对模型形成**类内共识**（"这两个都是 X，所以 X 是这种语义"）；减半后只剩单条，模型更容易摇摆到近义 label。**多样性 < 共识**。
**教训**：在小模型 + 细粒度 label 场景，"覆盖更多 label"反而稀释判别信号。

### 失败 4：二段式 LLM 调用（81.1% → 79.4%）

**架构**：Stage 1 用全 LABEL_SET 让 LLM 出 top-3 候选 → Stage 2 在 (top-3 ∪ 检索 top-K) 的 shortlist 内精选。
**预期**：求解空间从 77 收到 6，stage2 内部命中率 ↑ → +3-5%。
**实测**：-1.7%（prompt ×1.57, compl ×4.55）。
**归因**：Qwen3-8B 的错是**系统性错**而非采样错（与失败 1 同源）——top-3 漏召正确 label 的概率与 top-1 直接选错的概率高度相关。stage1 漏召时 stage2 必错，且无救济通道。检索补充的 label 也是同特征空间的 shortlist，多样性增量为零。
**教训**："多次询问 / 列候选 / 重排"这类 ensemble 思路对系统性错全部无效。

### 失败 5：LIKELY_CANDIDATES 高亮提示（81.1% → 78.8%）

**架构**：保留全 LABEL_SET，但在 user msg 里加一段 `LIKELY_CANDIDATES: <检索 top-K 的 label>` 给模型"高亮提示"。模型仍可选任意 label（不强制收紧）。
**预期**：基于 hard-case 分析中"被吸光 label 抢走"的发现（`card_arrival → card_delivery_estimate` 等近义混淆），通过软提示让模型注意检索强信号。+1-3%。
**实测**：-2.3%（prompt 713→765, compl 不变）。
**归因**：高亮列表里通常**包含**那些"吸光 label"（因为它们检索分数也高），软提示反而**强化**了被吸光的概率。冗余信息（few-shot examples 已经让 LLM 看到这些 label）让模型把 LIKELY 当成"答案空间"理解，而非"高优先候选"。
**教训**：当问题来自模型的内禀偏置（偏好通用 label）时，**任何把"候选 label"再次显式列出的 prompt 修改**都可能加重偏置而非纠正。要纠正需要反向信号（贬低吸光 label）或语义级区分（label 描述）。

### 失败 6：lazy 生成 label 语义描述塞进 prompt（81.1% → 79.4%）

**架构**：update 阶段不动；首次 predict 时 `_ensure_descriptions` 用 `threading.Lock` 守护，调 LLM **3 次批量生成** 77 个 label 的简短描述（每批 ~30 个 label）；之后所有 predict prompt 把 LABEL_SET 改写成 `label: description` 一行一个。
**实现**：`solution.py` 加 76 行，新增 `_ensure_descriptions / _build_descriptions / _gen_batch_desc / _parse_desc_response`。生成成功率 76/77。批量 prompt 体感合理（"User wants to know when a transfer will be processed" 准确刻画 `transfer_timing`）。
**预期**：直击 hard-case 病灶（吸光 label 偏好），让模型从语义边界判断而非 label 名关键词。+3-7%。这是探索清单 §四统一根因里"唯一未跑路径"。
**实测**：-1.7%（prompt 713→1410, compl 3.8→5.8, 时间 64s→86s）。
**归因**：

1. **prompt 长度翻倍稀释了 few-shot 信号**——模型注意力在 1410 token 的 label 描述堆里游走，对真正起作用的 8 条检索 example 关注下降。
2. **描述本身是 LLM 自生成的**，质量参差且**风格趋同**——它倾向于把多个相近 label 描述成同样模糊的"用户询问 X"，并未真正区分 `card_arrival` 和 `card_delivery_estimate`。
3. **吸光偏置不在"模型不懂语义"层面**——而是在"决策时偏好简短/通用 token"的解码偏置。给 description 不影响这层偏置。

**教训**：

- 任务清单 §六 §1 "Label → 原型"思路，**在 Qwen3-8B 这种规模上未验证**——它能让大模型变得更精确，但小模型反而被信息过载拖累。
- 要真正纠正吸光偏置，可能需要**反向措辞**（"避免选择通用 label 除非 query 明确通用"）或**强制选择**（受限解码到 shortlist），但前者已被失败 5 证伪，后者会牺牲泛化（私有集 OOD label 必错）。
- "唯一未跑的路径"现已跑完且失败——**81.1% 是 Qwen3-8B + 当前框架在 DEV 上的真实 ceiling，无悬念**。

## 三-bis、Hard case 错误结构分析（[hardcase_20260507_211414/summary.md](../output/hardcase_20260507_211414/summary.md)）

`102 错误 = 75 LLM 决策错（73.5%） + 27 检索 hard miss（26.5%）`

**检索 recall 真值**：

| 范围 | gold 命中率 |
|---|---|
| top-1 | 47.3% |
| top-5 | 76.8% |
| top-10 | 86.3% |
| top-24 | **92.9%** |

**错误样本中 gold 在哪里**：

| 状态 | 占错误 | 含义 |
|---|---|---|
| gold == top-1 | 13.7% | 检索把正确答案放第一位，LLM 仍偏不选 |
| gold ∈ top-5 | 31.4% | 强信号都没救 |
| gold ∈ top-10 | 50.0% | — |
| gold ∈ top-24 | **73.5%** | gold 在检索池内 → 纯 LLM 决策错 |
| gold ∉ top-24 | **26.5%** | 检索完全 miss，每 label 仅 3 条训练样本，hard miss 不可救 |

**核心发现：吸光 label 模式**（任务清单 §六 §1 提到的 "Label → 原型" 策略对应的痛点）：

| 模型最爱预测错的 label（"吸光"） | 错预次数 | 典型受害 gold |
|---|---|---|
| `card_delivery_estimate` | 8 | `card_arrival`, `order_physical_card` |
| `declined_card_payment` | 7 | `top_up_failed`, `virtual_card_not_working` |
| `transfer_timing` | 4 | `balance_not_updated_after_bank_transfer` |
| `exchange_rate` | 4 | `wrong_exchange_rate_for_cash_withdrawal` |

**最难的 gold label**（每 7 条 dev 错 4+ 条）：

| gold label | 错误率 |
|---|---|
| `wrong_exchange_rate_for_cash_withdrawal` | **86%** (6/7) |
| `topping_up_by_card` | 71% (5/7) |
| `supported_cards_and_currencies` | 71% (5/7) |
| `card_arrival` | 57% (4/7) |
| `get_physical_card` | 57% (4/7) |

**根本归因**：Qwen3-8B 对**短而通用**的 label 有先验偏好（"它们覆盖语义更宽，安全选项"），在面对细粒度近义类时倾向于回退到通用 label。这种偏置：

- 不是 prompt 加东西能修的（失败 5 验证）
- 不是 reasoning 能修的（失败 2 验证）
- 不是 ensemble 能修的（失败 1, 4 验证）
- **唯一可能修的**：给每个 label 配一句精确语义描述，把"通用 label 仅适用纯查询场景"等硬约束写进 prompt。但这要求 update 阶段调 LLM 生成描述，引入新预算与新失败面。

## 三-ter、跨部署实证（[api_determinism_probe.py](test/api_determinism_probe.py)）

**问题**：之前在 DashScope 上观察到"同 prompt 多次输出严格一致"（投票 N=3 token 完全等比），归因为"服务端 cache / 低温采样"。但任务清单 §一 §6 写"4 次采样取均值"，暗示评测方部署是**真采样**——这意味着投票在评测方下可能反而有效。需要换一个**已知真采样**的部署来验证。

**实验**：切到 SiliconFlow（独立 API 服务，相同 `Qwen/Qwen3-8B`）做对照。

| 测试 | DashScope | SiliconFlow |
|---|---|---|
| 同分类 prompt × 5 次输出 | 5/5 完全一致 | 5/5 完全一致 |
| 创意写作 × 3 次输出 | （未测） | **3/3 完全不同**（确证真采样） |
| 模糊分类 × 5 次（temp=1.0） | — | 5/5 一致 |
| 模糊分类 × 5 次（**temp=1.5**） | — | 5/5 一致 |
| 不同 `seed` × 5 次（同分类 prompt） | — | 5/5 一致 |
| **N=1 P1 baseline @ 539 dev** | **81.1%** | **79.4%** |
| **N=3 投票 @ 539 dev** | 81.1%（无变化） | **79.8%（+0.4%，~2 条）** |

**两个独立结论**：

1. **失败 1 的归因要修正**：投票无效**不是**因为 DashScope 服务端 cache，**而是 Qwen3-8B 在分类任务上 logits 极度 peaked**——首选 token softmax 概率接近 1.0，即使在已实证真采样的 SiliconFlow 上多次调用也几乎 always 选同一个。这是**模型本身的特性**，不是部署的特性。**评测方部署也救不了投票**，这条路彻底封死。

2. **跨部署有 ±1-2% 浮动**：同模型同 prompt，DashScope vs SiliconFlow 差 1.7%（10 条/539）。差异来自 fp 精度（fp16 vs bf16）、kernel 版本、sampling 实现等不可控因素。**评测方分数不能从本地数字外推**——80-83% 都在合理范围。

**方法论意义**：以后做"对模型行为下结论"的实验前，**先用 [api_determinism_probe.py](test/api_determinism_probe.py) 探一下当前服务的行为基线**——能避免把"服务特性"当成"模型特性"或反之。

## 四、统一根因（贯穿六次失败）

> Qwen3-8B 在该任务上的错误 **73.5% 是"模型看到正确 label 仍系统性选错"**（hard case 分析证实），剩余 26.5% 是检索 hard miss（每 label 仅 3 条训练样本的天花板）。
>
> 系统性错误具体表现为**偏好"短而通用"的吸光 label**（`card_delivery_estimate / declined_card_payment / transfer_timing / exchange_rate`），六次"加东西"实验依次证伪了所有标准技巧：
>
> - 不是采样抖动 → 投票无效（失败 1，跨部署实证 §三-ter 加固）
> - 不是 reasoning 缺失 → CoT 反作用（失败 2）
> - 不是 example 多样性不足 → 减半反而下降（失败 3）
> - 不是候选空间过大 → 收紧到 shortlist 也无救（失败 4）
> - 不是模型没"注意到"正确候选 → 高亮提示反而强化吸光（失败 5）
> - 不是模型不理解 label 语义 → 给精确描述也救不回（失败 6）
>
> 第六次失败把"唯一未跑的路径"也跑完了。结合 §三-ter 跨部署实证：**问题不在 prompt 信息层面也不在部署层面**（已验证 SiliconFlow 真采样下 N=3 仅 +0.4%），**而在模型本身的解码偏置**——Qwen3-8B 在多 token 输出时 logits 极度 peaked，倾向于选短的、出现频率高的 label。这种偏置不能用 prompt 修，要用 **constrained decoding**（受限解码到 shortlist）或换模型。前者会牺牲 OOD 泛化，后者评测方控制。

**结论**：在"检索 + full LABEL_SET + few-shot"框架下，**~80% 是 Qwen3-8B 在 DEV 上的真实 ceiling**（DashScope 81.1%、SiliconFlow 79.4%，跨部署±1-2%浮动）。继续 prompt engineering 是负 ROI，六次实验已**穷尽**任务清单 §五/§六 列出的全部建议方向。

## 六-bis、few-shot 数量 sweet spot 曲线（cap=2 固定）

> 连同历史实验，完整扫描了 `n = 4, 6, 8, 10, 12` 五点（n=0/2 因下行趋势已线性加重，ROI 极低故略）。所有点 base 相同（RRF + G parser），仅改 `_build_messages` 里 `len(chosen) >= N`。

```text
few-shot n     DEV 准确率    Δ vs n=8
   12          80.3%        -1.1%    (测试 I)
   10          81.1%        -0.3%    (Exp A，hard negative +2)
 ★ 8           81.4%        0        ← peak (当前 solution.py)
    6          79.0%        -2.4%
    4          78.3%        -3.1%

       曲线形状：严格单峰 (unimodal)，两侧均单调下降
       左侧斜率（减）≈ -0.4%/条 → 右侧斜率（加）≈ -0.2%/条
       左侧下降更陡 → 信号不足比信号稀释代价大
```

**核心发现**：

1. **n=8 不只是"减会退化"的下界，也是"加会退化"的上界**——**真 sweet spot**，双向敏感。
2. **左侧下降更陡**（每少 1 条 ≈ -0.4%）**> 右侧**（每多 1 条 ≈ -0.2%）——说明 **Qwen3-8B 对"类内共识"的需求 > "label 多样性"的容忍度**。
3. 这和失败 3（cap=1 倒序 -2.1%）机制一致，但方向完全相反（cap 控制**每 label 几条**，n 控制**总共几条**），两个维度独立验证了"**cap=2 × n=8 = 黄金组合**"。
4. **left-side clip 解释了 hard miss 26.5%**：如果一条 query 的 gold label 只出现在 top-10/20 里，n 越小越没机会进 few-shot → 解释为何 n=4 比 n=6 再掉 0.7%。

**报告价值**：这条曲线图是探索报告里最干净的"**边际收益递减 + 双向下降**"ablation study，比任何单点实验都直观——可以直接做正文核心图。

## 九、多任务实证（新造 OOD + MCQ mock 数据集）

> 数据来源：2026-05-08 手造 `train_ood.jsonl` / `test_ood.jsonl`（186/600 条，52 labels，覆盖 sentiment / news / questions / smart_home / travel / food 等跨域）、`train_mcq.jsonl` / `test_mcq.jsonl`（96/384 条，A/B/C/D 均衡，含 Python/生物/HellaSwag/教科书题）。
> 数据质量：label 覆盖 test⊆train 符合规则，文本长度分布真实（OOD min=4 max=662，MCQ min=84 max=2694），label 分布均衡。

### 9.1 三检索方法 × 三任务完整 ablation（cap=2, n=8, G parser 固定）

| 检索方法 | DEV (20%) | OOD (60%) | MCQ (20%) | 加权分 | 时间戳 |
|---|---|---|---|---|---|
| IDF-cosine | 81.1% | 78.8% | 78.1% | **79.1%** | 20260508_13xxxx cos |
| BM25-only | 81.4% | 79.2% | 80.7% | **79.9%** | 20260508_13xxxx bm25 |
| **RRF 融合** | **81.4%** | **80.3%** | **81.2%** | **🏆 80.7%** | 20260508_13xxxx rrf |

**关键结论**：

1. **RRF 全面占优**——三任务上都是第一或并列第一，加权 +1.6% vs cos / +0.8% vs BM25。
2. **DEV 上 BM25 = RRF 是假象**——DEV 文本短、同分布，是 best case；**OOD 和 MCQ 才是分化场景**。
3. **IR 文献里 RRF 的"分布偏移下鲁棒性"**在 OOD 上被**实证**——RRF 比 BM25 多 +1.1%，比 cos 多 +1.5%。
4. **cos 在长文本上显著差**——MCQ 题干最长 2694 char，cos 的 `|q|·|d|` 归一化被 char-gram 向量维度吹爆，BM25 的 `b=0.75` 文档长度归一化恰好修正这点（MCQ 上 BM25 比 cos 多 +2.6%）。
5. **RRF 保留决定被加固**——此前 DEV 上"BM25-only 和 RRF 平手"的消融给人"是否简化"的动摇，现在**OOD + MCQ 直接封死了 BM25-only 路线**。

### 9.2 分任务性能画像

| 任务 | 最佳 | prompt/条 | compl/条 | 耗时 | 备注 |
|---|---|---|---|---|---|
| DEV 539 条 | 81.4% | 712 | 3.9 | 66s | few-shot sweet spot 双向敏感（见 §六-bis） |
| OOD 600 条 | 80.3% | 553 | 2.4 | 71s | 跨域 label 检索 recall 下降但 RRF 有效兜住 |

### 9.3 加权分的实战意义

```text
RRF 加权 80.7% = 0.2 × 81.4 + 0.6 × 80.3 + 0.2 × 81.2
```

考虑到：
- 跨部署 ±1-2% 浮动（§三-ter）
- 评测方部署一致性（学院 vllm 自部署，应该不比 DashScope 更好）
- 私有集可能比 mock 更难（私有测试集不可见，我们的 OOD mock 是**自己造的，可能偏乐观**）

### 9.4 §六-bis few-shot 曲线在 OOD / MCQ 上是否仍然成立？（未验证）

DEV 上 n=8 是 sweet spot，左陡右缓。但 OOD 和 MCQ 的最优 n 可能不同：

- **OOD**：label 数 52 < 77，且 per-label ~3.6 条，可能 n=6 就够
- **MCQ**：label 只有 4 个，每 label 24 条 train，**n=8 对应最多 4×2=8 条**（cap=2 填满所有 label），prompt 已 1400 token，n 再大会撞预算

这是后续验证方向，但优先级排在下面"重跑失败 6"之后。

## 十、P0 实验：失败 6 在 OOD 上完全反转（+3.1% 加权）

> **这是整个探索过程中最大的单次突破**。读 FAQ 发现任务权重 20/60/20 后，按 §八-8.4 的猜想在 OOD 上重跑失败 6，结果三任务加权 80.7% → **83.8%（+3.1%）**。

### 10.1 变更内容

- `_ensure_descriptions()`：首次 predict 时按 10 labels/batch 调 LLM（OOD 6 批 ~15s，DEV 8 批 ~18s）
- `_should_use_descriptions()`：若单字符 label ≥50% 则判为选择题任务，**自动 skip** description 生成
- `_build_messages` 内 `build()`：有 descriptions 时 label section 从 `LABEL_SET: a, b, c` 改为多行 `label: meaning`；否则保持原单行格式
- 每个 label 用 3 条原始样本作 prompt 上下文，要求生成 <20 词的 "discriminative not tautological" 描述

### 10.2 三任务实证对比

| 任务 | 之前 (RRF+G) | **现在 (+desc)** | Δ | prompt 变化 | 备注 |
|---|---|---|---|---|---|
| DEV (20%) | 81.4% | **83.7%** | **+2.3%** | 712 → 1663 | 与失败 6 (-1.7%) 完全反向！ |
| OOD (60%) | 80.3% | **84.7%** | **+4.4%** | 553 → 1097 | 符合 §八-8.4 猜想 |
| MCQ (20%) | 81.2% | 81.2% | 0 | 1400 持平 | 自动 skip 生效 |
| **加权** | 80.7% | **🏆 83.8%** | **+3.1%** | — | — |

### 10.3 为什么这次赢了，而失败 6 输了？三重归因

> 这是报告的核心亮点。同样的"LLM 生成 label 描述"思路，DEV 上 **-1.7% → +2.3%** 完全反转。归因：

**原因 1：prompt 结构更好**

- 失败 6：`label: description` 简单粘贴进原 `LABEL_SET` 单行位置
- 现在：多行格式 `  label: meaning`，每 label 独占一行，用 `LABEL: name :: desc` 双冒号作解析锚点
- → LLM 更容易把"描述"和"label 名"在注意力上解离，不会把描述当 label 候选

**原因 2：描述质量更高**

- 失败 6：prompt 没强调"辨别性"
- 现在：明确要求 "discriminative not tautological"、"under 20 words"、"based on example utterances"
- → 生成的描述更锐利，每个 label 的语义边界更清晰

**原因 3（最关键）：base 从 cos-only 升级到 RRF+G**

- 失败 6：base = IDF-cos（81.1%），few-shot 召回噪声大；加入 description 稀释 few-shot 后，检索噪声没有信号兜底 → **净负**
- 现在：base = RRF（81.4%），检索本身更准；description 成为**锦上添花的语义信号**，和已经正确的 few-shot 叠加 → **净正**
- → 这是"**底层先做对，上层再加层**"的经典案例。基础组件质量决定增强策略的方向能否反转

### 10.4 一般性教训（报告金句）

> **失败不是终点，是假设。每个失败实验都应记录"若 X 改变，结论可能反转"的条件。**

失败 6 原归因是"Qwen3-8B 规模限制"——但真正的限制是 base 的检索噪声 + prompt 结构拙劣。**换了 base 换了格式，同一个机制立刻从 -1.7% 翻成 +2.3%**。这是对 §四统一根因"模型解码偏置封死 ceiling"的**重要修正**：

- ceiling 是存在的，但**不是 81.4%**
- 之前以为 81.4% 是"prompt engineering 触底"，其实只是"在当时 base 的触底"
- 升级 base 后，同一个 prompt-engineering 手法能再拿 +3.1%

### 10.5 对统一根因章节 §四 的修正

§四当时的结论："问题在模型解码偏置,不在信息层——任何加东西都没用"。

**现在修正为**：

1. DEV 同分布场景下，模型本身偏置确实是主要瓶颈，因为 few-shot 已经给了足够强的 in-context 信号
2. **OOD 场景下，信息层至关重要**——few-shot 因 lexical mismatch 失效，label 描述成为**唯一可用的语义信号**，收益巨大
3. 失败 1-6 全部在 DEV 上做的实验，其"都失败"的共同根因不是"加东西没用"，而是 **"在 DEV 这个 in-distribution 场景下，加的信息和 few-shot 冗余"**

### 10.6 其他可能受益的失败实验（待后续验证）

参考 §八-8.4 的表格，以下失败实验应该**在 OOD 上重跑一次**：

| 实验 | DEV | OOD 猜想 | 优先级 |
|---|---|---|---|
| 失败 2（reasoning 二段） | -2.1% | OOD 需要推理链路，可能翻正 | P1 |
| 失败 4（二段式候选） | -1.7% | 同理，shortlist 后再选 | P1 |
| 失败 5（LIKELY 高亮） | -2.3% | 若 LIKELY 来自 RRF 更准，可能有效 | P2 |
| 测试 II（synthetic 样本） | 0% / 3x 耗时 | OOD 覆盖不足时可能救场 | P2 |

这些都可以按同样的"OOD mock 重跑"流程处理。

## 十一、P0.2 实验：失败 2（reasoning）的任务异质性 + 任务自适应路由（+0.58% 加权）

> **继 §十 之后第二大发现**：把 reasoning 在三任务上单独测试，揭示**强烈的任务异质性**；再用 `_descriptions` 状态作信号实现任务自适应路由，加权 83.8% → **84.4%**。

### 11.1 三任务单独测 reasoning 的异质性

统一开启 reasoning（让模型先给 ≤25 词 reasoning 再 `Label: <xxx>`）：

| 任务 | 权重 | 无 reasoning（P0 baseline） | 开 reasoning | Δ |
|---|---|---|---|---|
| DEV | 20% | 83.7% | **84.8%** | **+1.1%** |
| OOD | 60% | 84.7% | 83.2% | **-1.5%** |
| MCQ | 20% | 81.2% | 84.6% | **+3.4%** |
| **加权（统一）** | — | 83.80% | 83.80% | **0%（完美抵消）** |

**核心发现**：reasoning 的收益**严重任务依赖**：

- **MCQ +3.4%**：选择题天然需要推理链路（题干→选项评估→决策），这是 Qwen3-8B 标准 CoT 受益场景
- **DEV +1.1%**：同分布细粒度分类，reasoning 帮助模型对照 few-shot 做微调（例如 `card_arrival` vs `card_delivery_estimate` 的边界思考）
- **OOD -1.5%**：这是反直觉但可解释的——OOD 下 **label 描述已经提供强语义信号**，reasoning 让 Qwen3-8B 产生**与描述冲突的自我分析**，回到失败 2 "小模型 CoT 反作用"的机制。**语义信号冗余比信号缺乏更危险**。

### 11.2 任务自适应路由（最终配置）

既然 reasoning 的收益 **和 description 使用状态互斥**：

```python
use_reasoning = not self._descriptions  # 无描述 = MCQ → 开 reasoning
                                         # 有描述 = DEV/OOD → 关 reasoning
```

这个 gate 是**零成本设计**——`_descriptions` 的状态在 `_ensure_descriptions()` 里已按单字符 label 自动分叉，reasoning 路由复用了这个信号。

**最终三任务成绩**：

| 任务 | 配置 | 结果 |
|---|---|---|
| DEV | desc 开、reasoning 关 | **83.9%** |
| OOD | desc 开、reasoning 关 | **84.2%** |
| MCQ | desc skip、reasoning 开 | **85.4%** |
| **加权** | 自适应 | **🏆 84.4%（+0.58% vs P0）** |

### 11.3 原因 3 的修正（比 §十 的三重归因更深）

§十 总结失败 6 反转的"三重归因"里提到的"底层先做对,上层再加层"。**失败 2 的反转验证了第四重原因——信号互斥性**：

> **两个增强 A 和 B 各自在 base 上都有收益，但 A + B 一起上不一定叠加，甚至可能相互抵消。**

- **DEV + OOD**（有 desc）：desc 提供强语义，再叠 reasoning → OOD 反而稀释（-1.5%）
- **MCQ**（无 desc）：没有语义信号，reasoning 填补空白 → +3.4%

→ **增强不可无脑叠加**。工程上需要"**互斥路由**"设计：基于任务特征选一个或多个最合适的增强路径，不是全部叠加。

### 11.4 对 §四 统一根因的第二次修正

§十 已经把"81.4% 是 ceiling"修正为"底层质量决定上层可能性"。§十一 再修正为：

1. **Ceiling 是配置敏感的**——同样的 Qwen3-8B，不同任务在不同增强组合下达到的 ceiling 差异巨大（DEV 81.4% → 84.8% / OOD 80.3% → 84.7% / MCQ 81.2% → 85.4%）
2. **没有单一"全局最优"配置**——任务自适应（路由到不同增强组合）才是现实
3. **失败实验的价值随 base 变化**——早期 base 下的"失败"在更强 base 下可能反转，或只在特定任务子集上有效

### 11.5 未跑的失败实验（更新 §十.6 后的状态）

| 实验 | DEV 原始 | OOD 实证 | 决策 |
|---|---|---|---|
| 失败 6（label description） | -1.7% | ✅ **P0 反转 +4.4%** | 已采纳 |
| 失败 2（reasoning） | -2.1% | 部分反转（MCQ+3.4%、DEV+1.1%、OOD-1.5%） | ✅ **P0.2 自适应路由采纳** |
| 失败 4（二段式候选 shortlist） | -1.7% | 未测 | P2 |
| 失败 5（LIKELY 高亮） | -2.3% | 未测 | P2 |
| 测试 II（synthetic 样本） | 0% / 3x 耗时 | 未测 | P3 |

失败 4 / 5 在当前 base（RRF + desc + adaptive reasoning）下重跑可能也会出现异质性。但**优先级已下降**——当前 84.4% 已很高，继续深挖的 ROI 小于写报告。

## 十二、P0.3 实验：朋友版本启发的"stopword + label 名 token 加权"— 失败，揭示"信号空间竞争"

> 对比朋友版本（FriendV = 单 BM25 + stopword + label-token 弱监督 + cap=3，加权 78.3%）后，尝试吸收其两个设计亮点叠加在我们 weighted-best 上：stopword 过滤 + label 名 token 加权。**全任务退步 -1.96% 加权**。

### 12.1 实验设计

- **stopword 过滤**：在 `_featurize` 里剔除 32 个英文 stopword（`the/a/my/your/is/are/...`），仅作用于 word 级特征，char 3/4-gram 保留
- **label 名 token 加权**：在 `update` 里对每个 label 名走 _featurize（拆下划线/问号）提取 word token 集合，存 `_label_name_tokens[label]`；在 `_search_bm25_ranked` 里，对 query ∩ label 名 token 的 doc 加 `0.5 × sum(idf)` 的 additive bonus

### 12.2 结果

| 任务 | 权重 | weighted-best baseline | P0.3 组合 | Δ |
|---|---|---|---|---|
| DEV | 20% | 83.9% | 82.2% | **-1.7%** |
| OOD | 60% | 84.2% | 81.7% | **-2.5%** |
| MCQ | 20% | 85.4% | 84.9% | -0.5% |
| **加权** | — | 84.4% | **82.4%** | **-1.96%** |

### 12.3 归因：信号空间竞争

朋友的 base 没有 LLM 生成 desc，**stopword + label-token 是他/她获取语义信号的主要手段**；我们的 base 已经有 desc，信号覆盖更全，再叠这两个增强会冲突：

**机制 1（stopword 过滤反噬）**：banking 客服场景里 `my/your` 区分用户自己的卡还是商户的卡，有语义信息。过滤掉 stopword 使多个近义 label 边界变模糊。→ DEV -1.7%

**机制 2（label-token boost 与 desc 竞争）**：
- query "I haven't received my new card yet" 含 "card"
- label-token boost 把所有含 "card" 的 label（card_arrival / card_linking / card_not_working...）的 doc 同时拉上 top-24
- cap=2 × 8 条 few-shot 被同质化（6 条都是 "card 相关"）
- 本来 desc 能区分 card_arrival "未收到卡" vs card_linking "绑卡问题"，现在 few-shot 信号让 LLM 困惑 → OOD -2.5%

**OOD 退步最严重（-2.5%）恰好印证了假说**——OOD 是 desc 收益最大的场景，叠加词法 boost 冲突最严重。

### 12.4 教训：**增强组件应该在"信号空缺维度"上工作**

> 设计良好的系统里，每个新增组件应该在一个"之前空缺"的信号维度上工作。
> 如果新组件和已有组件争夺同一信号空间，它只会把原本清晰的信号抖乱。

- **我们加入 desc（§十）**：填补"LLM 无法从 few-shot 推出语义边界"的空缺 → 收益大
- **我们加入 adaptive reasoning（§十一）**：填补"MCQ 任务需要推理链路"的空缺 → 收益大
- **吸收朋友 stopword**：**和 desc 争语义空间**（都在告诉 LLM "哪些 label 候选"）→ 抵消
- **吸收朋友 label-token boost**：**和 desc 争语义空间**（都在给 LLM 加标签偏好）→ 抵消

朋友的 trick 在 FriendV 的朴素 base 上是对的（填补无 desc 空缺），**迁移到我们的富 base 就从正确变错误**。

### 12.5 对"朋友版本可吸收点"的更新结论

| 朋友版本设计点 | 是否值得吸收 | 理由 |
|---|---|---|
| Stopword 过滤 | ❌ 否 | 和 desc 冲突,损失 banking 细粒度区分 |
| Label 名 token 加权 | ❌ 否 | 和 desc 冲突,同质化 few-shot |
| cap=3(vs 我们 cap=2) | ❌ 否 | 失败 3 已证伪 cap=1 负向，cap=3 方向未扩大验证但预期同样负向（多样性过剩稀释） |
| \`\`\` 包裹 query | ❌ 否 | XML 包裹 `<query>` 抗注入更稳（mock 9/9 证） |
| System msg 简短 | ❌ 否 | 失败 A3 已证伪 |

**结论:朋友版本没有我们能单独吸收的组件**——他/她的设计优点在他/她的朴素 base 上是最优的,迁移到我们的 base 上全部失效。这本身是对"富 base → 组件搭配"思路的强证据。

## 十三、P0.4 实验：MCQ 专用 system_msg 与 "3-way 分支" 重构

> 继 §十、§十一 后的第三次关键改进。**MCQ +2%，weighted 84.4 → 84.5**，并借机修复了 bak_best_dev 的语义耦合 bug。

### 13.1 MCQ few-shot 数量扫描（失败）

观察到 MCQ 的 A/B/C/D label **语义随题而变**，猜想 few-shot 是噪声，应减少。实测：

| MCQ 配置 | 准确率 | 备注 |
|---|---|---|
| n=4（half cap=2） | 83.1% | -2.3%，示例不足，LLM 失去"题→选项→答"的任务框架 |
| **n=8 cap=2（baseline）** | **85.4%** | 当时最好 |
| n=12 cap=3 | 83.6% | -1.8%，多样性过剩 |

**Sweet spot 在 n=8**——与 DEV 一致（§六-bis 同样显示 n=8 是双向 peak）。MCQ 即使 few-shot 看似"语义无用"，**结构锚定**作用不可替代。

### 13.2 MCQ 专用 system_msg（+2% 稳定）

原 system_msg 对所有任务都是"You are a strict text classifier"。对 MCQ 任务来说，**任务定义错位**——Qwen3-8B 被当成普通分类器，没有进入"解题"心智模型。

改为 MCQ 专属 system_msg：

> "You are solving a multiple-choice question. The input inside <query> contains a question and its options; choose the correct option label from LABEL_SET. First give ONE short sentence of reasoning..., then output your final answer on a new line as `Label: <exact label>`."

**MCQ 从 85.4% → 87.0-87.8%**（多次跑稳定 +2%）。一句任务框架的改动换 9 条翻正。

### 13.3 Refactor：`system_msg` 从 2 分支改为 3 分支

原逻辑（2 分支）：

```python
use_reasoning = not self._descriptions
if use_reasoning:
    system_msg = "MCQ solver..."  # 错！use_reasoning=True 的 DEV 也会套 MCQ msg
else:
    system_msg = "classifier..."
```

潜在 bug：`bak_best_dev` 用 `use_reasoning = True` 强制全任务开 reasoning，这样 DEV 也会被当成选择题 → **语义错位**。

改为 3 分支：

```python
mcq_mode = not self._descriptions        # 独立信号
use_reasoning = mcq_mode                  # weighted：仅 MCQ；dev：强制 True
if mcq_mode:            system_msg = "MCQ solver + reasoning"
elif use_reasoning:     system_msg = "classifier + reasoning"
else:                   system_msg = "classifier direct"
```

**同时修复了 bak_best_dev 的正确性** + **让 MCQ 专属 msg 只在真 MCQ 场景生效**。

### 13.4 两版最终指标（refactor 后）

| 备份文件 | DEV | OOD(mock) | MCQ(mock) | **加权** |
|---|---|---|---|---|
| `bak_best_weighted`（当前默认） | 83.7% | 83.5-84.2% | **87.0-87.8%** | **🏆 84.5%** |
| `bak_best_dev` | **85.2%** | 81.8% | 86.7% | 83.4% |

### 13.5 元教训

§十：「底层做对，上层能反转」。
§十一：「增强不能无脑叠加，需任务路由」。
**§十三：「任务框架错位和信号缺位一样致命——给 LLM 正确的任务定义和给它正确的语义信号同等重要」。**

MCQ system_msg 的 +2% 不来自任何新信息，**只来自"告诉 LLM 它在干什么"的一句话**。这是对 §四 "Qwen3-8B 解码偏置封死 ceiling" 结论的第三次修正：

1. §十 修正：**底层 base 升级后，失败可反转**
2. §十一 修正：**同一任务上不同增强的最优路由是任务异质的**
3. §十三 修正：**任务定义错位会让模型不使用已有能力；一句明确的框架描述就能解锁**
