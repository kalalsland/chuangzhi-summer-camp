| 指标 | 值 | 备注 |
|---|---|---|

| 抗注入 mock | **9/9 全挡** | 9 种攻击向量 |

### 三段式演化

**阶段 I（P1 基线）：DEV 0% → 81.1%**（检索 few-shot + LABEL_SET + 抗注入封装），跨部署 ±1-2% 浮动。

**阶段 II（八次"加东西"失败 + 四次反转）**：
- 8 次 prompt engineering 实验**全部 DEV 回归**（最大 -2.3%，最小 -0.4%），在当时 base 下穷尽任务清单 §五/§六 全部建议
- **FAQ 公布三任务权重 20/60/20 后**重新审视失败实验 → 其中 3 次在 OOD/MCQ 上**完全反转**：
  - **§八 RRF 融合**：cos + BM25 融合 → 加权 80.7%（+1.6 vs cos-only）
  - **§九 LLM label 描述**（§5.6 失败 6 反转）：DEV +2.3% / OOD +4.4% → 加权 83.8%
  - **§十 任务自适应 reasoning**（§5.2 失败 2 部分反转）：MCQ +3.4% / DEV +1.1% / OOD -1.5% → 自适应路由加权 84.4%
- **§十.5 MCQ 专用 system_msg**：把任务框架从"分类器"改为"解题"，MCQ +2% → 加权 84.5%

**阶段 III（鲁棒性加固）**：FAQ 细读后发现两个高风险盲区并修复：
- **§十二 P0**：MCQ 检测启发式改用 train text 选项 stem 扫描（代替脆弱的"单字符 label 比例"）—— mock 三任务 0%/0%/100% 分离完美
- **§十二 P1**：LLM 异常兜底从"最高频 label"升级到"检索 top-1"

### 最终决策：`use_reasoning = True` 全任务开推理

三个候选变体（均带 P0/P1 加固）4-run 对比：

| 变体 | DEV | OOD(mock) | MCQ(mock) | 加权(mock) |
|---|---|---|---|---|
| `use_reasoning = mcq_mode`（加权最优 mock） | 83.6% | 82.7% | 87.5% | 83.84% |
| **`use_reasoning = True`（最终提交）** | **85.2%** | 82.0% | 87.2% | 83.68% |

三变体今日加权分差 0.34% 在噪声带内。**DEV 是官方唯一可信代理**——mock OOD/MCQ 可能偏乐观，不值得为 mock 上的 +0.16% 牺牲 DEV 上 +1.6% 的实打实置信度。

### 核心结论

1. **失败不是终点，是假设**：每个失败实验都有"若 X 改变结论可能反转"的隐含条件。失败 6 在 RRF + 新 prompt 格式 base 上从 -1.7% 翻成 +2.3%，失败 2 在任务路由下从 -2.1% 分解成"DEV +1.1% / OOD -1.5% / MCQ +3.4%"的异质性。
2. **底层质量决定上层可能性**：同样的 label 描述机制，在 cos-only base 上是信号稀释，在 RRF base 上是语义补强。
3. **增强不能无脑叠加**：朋友版本的 stopword + label-token boost（§十三）在他的朴素 base 上是最优，迁移到我们富 base 全部退步 -2%，因为**信号空间竞争**。
4. **任务框架错位和信号缺位一样致命**：MCQ system_msg 的 +2% 不来自任何新信息，只来自"告诉 LLM 它在干什么"这一句话。

## 一、任务理解与硬约束

任务给定 `MyHarness` 接口（`update(text, label)` + `predict(text)`），要求：

- **训练流外部记忆**：先吃完 `train_dev`，建索引，但不能存到磁盘
- **推理时**：仅传 `text`，返回与 gold label **exact match** 的字符串
- **库白名单**：`stdlib + numpy + harness_base`（不准 sklearn/torch/openai）
- **Token 预算**：单次 `call_llm` prompt > 2048 会被尾部截断（[run.py:30-56](run.py#L30-L56)）
- **泛化**：私有集还有 OOD + A/B/C/D 选择题 + Prompt Injection 样本
- **稳定性**：4 轮采样取均值，输出抽不稳直接掉分

**红线**：不能"抛开 LLM 用 TF-IDF 当分类器"——选择题必须由 LLM 决策。

## 二、最终 Harness 设计

### 2.1 模块分层

```
┌──────────────────────────────────────────────────────────────┐
│  update(text, label)                                         │
│      ├─ self.memory.append((text, label))   [基类]            │
│      ├─ self._labels / self._label_set       (label 集合管理)  │
│      ├─ Counter(_featurize(text))            (TF 向量)         │
│      ├─ inverted index: token → [doc_idx]    (倒排, O(1) 召回) │
│      ├─ self._doc_len[idx]                   (BM25 用文档长度) │
│      ├─ self._total_len (avgdl 增量)          (BM25 归一化)    │
│      └─ self._df[token] += 1                 (document freq)  │
└──────────────────────────────────────────────────────────────┘
                          │
                          ▼
┌──────────────────────────────────────────────────────────────┐
│  predict(text)                                               │
│      ├─ 0. 首次触发: _ensure_descriptions() 批量生成 label 描述 │
│      │     (非 MCQ 任务, 自动路由由 _detect_mcq_task 判定)      │
│      ├─ 1. 检索: RRF 融合 (IDF-cos + BM25) over 倒排索引       │
│      │      → top-24 候选 doc                                 │
│      ├─ 2. per-label cap=2, 取最多 8 条 few-shot              │
│      ├─ 3. Prompt 构造（4 级 token 预算降级, 三分支 system_msg）│
│      │     ┌─ system: MCQ / 分类+reasoning / 分类 direct 三选一│
│      │     ├─ LABEL_SET: 多行 "label: meaning" (非MCQ) 或单行  │
│      │     ├─ few-shot: <ex><text>...</text><label>X</label> │
│      │     └─ <query><text>...</text></query>                │
│      ├─ 4. self.call_llm(messages)                           │
│      │     (失败时走 _retrieval_fallback → 检索 top-1 label)   │
│      └─ 5. _parse_label(resp) [5 级解析归一化]                  │
│            exact match → JSON → 单字符 → 子串 →                │
│            Levenshtein ≤ 2 → Jaccard 最近邻                   │
└──────────────────────────────────────────────────────────────┘
```

### 2.2 关键决策

| 决策 | 取值 | 理由 / 实证来源 |
|---|---|---|
| 检索特征 | char-3gram + char-4gram + word (unicode) | 短文本 query 需字符级容错；word 维度处理同义同根 |
| **检索算法** | **RRF 融合（IDF-cos + BM25）top-60** | 三任务消融（§八）：RRF 加权 80.7% > BM25 79.9% > cos 79.1%。OOD 上 RRF 比 cos 多 +1.5%（分布偏移场景下 RRF 鲁棒性实证） |
| 检索召回 | top-24 | gold ∈ top-24 = 92.9%（§四 hard case 分析） |
| Few-shot 选样 | per-label cap=2，最多 8 条 | cap=1 实测 -2.1%；双向 n 扫描曲线 n=8 是严格单峰 sweet spot（§六-bis） |
| **LABEL_SET 呈现** | **非 MCQ 任务：多行 `label: description`（LLM 生成）；MCQ：单行逗号分隔** | 失败 6 反转（§九）：OOD 下 +4.4%，DEV 下 +2.3% |
| **System msg** | **3 分支：MCQ 解题 / 分类+reasoning / 分类 direct** | §十.5 MCQ system_msg +2%（"告诉 LLM 在干什么"的一句话换 9 条翻正） |
| **Reasoning 开关** | **全任务强制 `use_reasoning = True`** | DEV +1.6% vs `mcq_mode` 变体。DEV 是官方测试集，选最高 DEV 置信度 |
| **MCQ 检测** | **`_detect_mcq_task()`：扫 train text 选项 stem（`\nA./\nA:/\nA)` 等），≥50% 命中判 MCQ** | §十二 P0 修复：比"单字符 label ≥ 50%"启发式更鲁棒，mock 三任务 0%/0%/100% 分离完美 |
| 抗注入 | 双层：system 声明 + `_parse_label` 强制白名单 | mock 9/9 攻击全挡 |
| 输出归一化 | 5 级 fallback（exact → JSON → 单字符 → 子串 → Levenshtein≤2 → Jaccard 最近邻） | 处理：reasoning+Label: 格式、单字符 A/B/C/D、拼写小错等 |
| **兜底** | **LLM 异常 → 检索 top-1 label**（而非最高频） | §十二 P1 修复：检索 top-1 命中 47.3% > 最高频的 ~5% |
| Token 预算 | 4 级降级（全 label+全 ex → 全 label 减 ex → 收缩 label → 仅 query） | DEV 1697 / OOD 1121 / MCQ 1426，远低于 2048 上限 |

|---|---|---|---|
| 准确率 | **85.2%** | 82.0% | 87.2% |

### 3.2 阶段 I：P0 → P1 的唯一杠杆

**Baseline 真的是 0%**（不是看错）：裸 zero-shot 让 LLM 直接生成 `direct_debit_payment_not_recognised` 这种 snake_case label 是不可能的——模型不知道候选集，平均只输出 2 个 token，几乎全部 exact-match miss。

**P1 一击解决**：把 `update()` 收集的 label 集合显式列入 prompt + 检索 8 条 (text, label) 作为 few-shot 锚定 → 模型有了候选集和模仿对象，准确率直接到 81.1%。

### 3.3 阶段 II 的战略转折点：FAQ 公布任务权重

读 FAQ 行 269/276 得知**任务权重为 20/60/20**（DEV/OOD/MCQ），**OOD 才是 60% 主战场**。此前 8 次失败全部在 DEV 上做的单任务实验，其"都失败"的共同根因不是"加东西没用"，而是 **"在 DEV 这个 in-distribution 场景下，加的信息和 few-shot 冗余"**。

于是在 OOD/MCQ mock 上重跑失败实验，发现：
- **失败 6（label 描述）完全反转**：DEV +2.3% / OOD +4.4%（见 §九）
- **失败 2（reasoning）部分反转**：MCQ +3.4% / DEV +1.1% / OOD -1.5%，强烈任务异质性（见 §十）

这是 Qwen3-8B 在该任务上**真正的 ceiling 跃升**，从 §六 原先宣称的 "81% ceiling" → 加权 84.5% → DEV 85.2%。

## 四、Hard Case 错误结构分析

### 4.1 方法

写 [test/hard_case_analysis.py](test/hard_case_analysis.py) probe，跑完整 539 条 dev，每条记录：检索 top-24 + LLM 预测 + 对错。输出 [output/hardcase_20260507_211414/](../output/hardcase_20260507_211414/)。

### 4.2 错误分解

```
102 错误 (18.9%)
├─ 75 (73.5%) — gold ∈ top-24 → 检索找到了，LLM 没选
│   ├─ 14 (13.7%) — gold == top-1（最强信号都没救）
│   ├─ 32 (31.4%) — gold ∈ top-5
│   └─ 51 (50.0%) — gold ∈ top-10
└─ 27 (26.5%) — gold ∉ top-24 → 检索 hard miss（每 label 仅 3 条训练，不可救）
```

**两类错误源独立**：
- **检索 hard miss（5%）**：每 label 仅 3 条训练样本的天花板
- **LLM 决策错（14%）**：模型的内禀偏置

### 4.3 "吸光 label" 现象

模型最爱预测错的 label 与最难的 gold 形成对应关系：

| 模型最爱选错的 label（"吸光"） | 错预次数 | 典型受害 gold |
|---|---|---|
| `card_delivery_estimate` | 8 | `card_arrival`, `order_physical_card` |
| `declined_card_payment` | 7 | `top_up_failed`, `virtual_card_not_working` |
| `transfer_timing` | 4 | `balance_not_updated_after_bank_transfer` |
| `exchange_rate` | 4 | `wrong_exchange_rate_for_cash_withdrawal` |

**最难的 gold label**：

| Gold label | 错误率 |
|---|---|
| `wrong_exchange_rate_for_cash_withdrawal` | **86%** (6/7) |
| `topping_up_by_card` | 71% (5/7) |
| `card_arrival` | 57% (4/7) |
| `get_physical_card` | 57% (4/7) |

**模式确认**：模型偏好**"短而通用"** label——`exchange_rate` 抢走 `wrong_exchange_rate_for_cash_withdrawal`、`card_delivery_estimate` 抢走 `card_arrival`。语义上"特定情形"被判成"通用情形"。

## 五、八次"加东西"实验的归因（每条都教训独立）

> 任务清单 §五 P4 明确写"**失败案例 + 归因，加分项**"。下面每个失败都附独立教训。
>
> **⚠️ 重要上下文**：本节所有实验 2026-05-07 在 **DEV 单任务** base（cos-only 检索）上做。FAQ 公布三任务权重 20/60/20 后，§六.2 对其中几个失败做了阶段 II 反转：**失败 6** 在 OOD 上翻成 +4.4%（§九），**失败 2** 拆成任务异质性路由（§十）。阅读时应把本节视为"Phase I 的证伪"而非"终结性结论"。

### 5.1 失败 1：自一致性投票 N=3（DashScope 81.1% → 81.1%，token ×3）

**预期**：单次抖动错的 query 通过多数票纠正 → +3-5%。
**实测（DashScope）**：分数完全不变。`prompt × 3.00, compl × 3.00` 严格等比 → 三次回答 token 数完全相同。
**初步归因**：服务端 deterministic（疑似 cache）。
**归因修正**（详见 §七 跨部署实证）：通过 SiliconFlow 独立 API 验证后修正为：**Qwen3-8B 在分类任务上 logits 极度 peaked**，即使在已知真采样的部署上也几乎 always 选同一个。这是**模型本身的特性**，不是部署的特性。
**教训**：投票需要"决策边缘抖动"前提；Qwen3-8B 在结构化分类上极度自信（无论选对还是选错），从源头无效。

### 5.2 失败 2：reasoning + label 二段输出（81.1% → 79.0%）

**预期**：让模型先 short reasoning 再出 label，强迫"对照 example 与 LABEL_SET 思考"，对近义混淆有效 → +3-5%。
**实测**：-2.1%。compl 从 3.8 涨到 21.3，模型确实在写 reasoning。
**归因**：Qwen3-8B 是小模型，**reasoning 反作用**——它直觉对的题被错误推理带偏（"This text mentions 'charge' so..." → 选 `extra_charge_on_statement` 而非 `direct_debit_payment_not_recognised`）。CoT 收益的临界规模通常在 30B+。
**教训**：CoT 不是普适增强；小模型上验证后再用。

### 5.3 失败 3：per-label cap=2 → 1 + few-shot 倒序（81.1% → 79.0%）

**预期**：cap=1 让 8 条 example 覆盖 8 个不同 label（多样性 ↑）；倒序让最相关 example 紧贴 query（recency bias）→ +1-3%。
**实测**：-2.1%。
**归因**：每 label **双 example 是 sweet spot**——"两条同 label 的 example"形成**类内共识**信号；减半后只剩单条，模型更容易摇摆到近义 label。**多样性 < 共识**。
**教训**：在小模型 + 细粒度 label 场景，"覆盖更多 label"反而稀释判别信号。

### 5.4 失败 4：二段式 LLM 调用（81.1% → 79.4%）

**架构**：Stage 1 全 LABEL_SET 让 LLM 出 top-3 候选 → Stage 2 在 (top-3 ∪ 检索 top-K) 的 shortlist 内精选。
**预期**：求解空间从 77 收到 6 → +3-5%。
**实测**：-1.7%（prompt ×1.57, compl ×4.55）。
**归因**：模型的错是**系统性错**而非采样错——top-3 漏召正确 label 的概率与 top-1 直接选错的概率高度相关。stage1 漏召 → stage2 必错，无救济通道。
**教训**："多次询问 / 列候选 / 重排"这类 ensemble 思路对系统性错全部无效。

### 5.5 失败 5：LIKELY_CANDIDATES 高亮提示（81.1% → 78.8%）

**架构**：保留全 LABEL_SET，但在 user msg 加一段 `LIKELY_CANDIDATES: <检索 top-K labels>` 软提示，模型仍可选任意 label。
**预期**：基于 hard-case 分析中"被吸光 label 抢走"的发现，软提示让模型注意检索强信号 → +1-3%。
**实测**：-2.3%。
**归因**：高亮列表里通常**包含**那些"吸光 label"（它们检索分数也高），软提示反而**强化**了被吸光的概率。冗余信息让模型把 LIKELY 当成"答案空间"理解，而非"高优先候选"。
**教训**：当问题来自模型的内禀偏置（偏好通用 label）时，**任何把"候选 label"再次显式列出的 prompt 修改**都可能加重偏置而非纠正。

### 5.6 失败 6：lazy 生成 label 描述（81.1% → 79.4%）

**架构**：update 不动；首次 predict 时用 `threading.Lock` 守护，调 LLM 3 次批量生成 77 个 label 的描述（每批 ~30 个）；之后所有 predict prompt 把 LABEL_SET 改写成 `label: description` 一行一个。
**预期**：直击吸光 label 病灶，让模型从语义边界判断而非 label 名关键词 → +3-7%（任务清单 §六 §1 "Label → 原型"思路）。
**实测**：-1.7%（prompt 713→1410, compl 3.8→5.8）。
**归因**：
1. **prompt 长度翻倍稀释 few-shot 信号**——模型注意力在 1410 token 描述堆里游走
2. **描述本身是 LLM 自生成的**，质量参差且**风格趋同**（多个相近 label 被描述成同样模糊的"用户询问 X"）
3. **吸光偏置不在"模型不懂语义"层面**——而是在"决策时偏好简短/通用 token"的解码偏置层面，给 description 不影响这层偏置

**教训**："Label → 原型"思路在 30B+ 大模型上有效，**在 8B 上反而被信息过载拖累**。

### 5.7 失败 7：A3 system msg 极简化（81.1% → 80.7%）

**改动**：system 从 5 句 (~60 token) 简化到 2 句 (~25 token)，去掉"no quotes/markdown/explanation"格式约束。
**实测**：-0.4%，compl 从 3.8 涨到 4.0。
**归因**：原 system 看似冗余的"no quotes/markdown/explanation"实际是有效约束——简化后模型偶尔加修饰（"\`label_xyz\`"、"label is X"），导致 parse 偶尔走 fallback 出错。
**教训**：当前 P1 prompt 的每个细节都被前面失败试错"硬化"过——**自然演化的 sweet spot**，看似冗余的约束都在起作用。

### 5.8 跨部署 N=3 投票 @ SiliconFlow（79.4% → 79.8%）

**关键证据**：用 SiliconFlow 独立 API 跑 N=3 投票，539 条 dev 仅 +0.4%（≈2 条变化），但创意写作 3 次完全不同——证明 SF **真采样**且分类任务上模型仍 effective deterministic。
**结论**：**模型 logits peaked** 是失败 1 的真因，跟部署无关。这条路彻底封死。

## 六、统一根因的三次修正（阶段 I → II → III）

### 6.1 阶段 I 原始版本（DEV 单任务视角，2026-05-07）

> Qwen3-8B 在 DEV 上 19% 错误中：
> - **73.5% 是"模型看到正确 label 仍系统性选错"**（gold ∈ top-24 但 LLM 不选）
> - **26.5% 是检索 hard miss**（每 label 仅 3 条训练样本的天花板）

> 系统性错误具体表现为**偏好"短而通用"的吸光 label**（§四）。八次"加东西"实验依次证伪了所有标准技巧：
>
> | 假设 | 证伪实验 |
> |---|---|
> | 不是采样抖动 | 失败 1 + §七 跨部署实证 |
> | 不是 reasoning 缺失 | 失败 2 |
> | 不是 example 多样性不足 | 失败 3 |
> | 不是候选空间过大 | 失败 4 |
> | 不是模型没"注意到"正确候选 | 失败 5 |
> | 不是模型不理解 label 语义 | 失败 6 |
> | 不是 system msg 太啰嗦 | 失败 7 |

**原始结论（已过时）**：在"检索 + 全 LABEL_SET + few-shot"框架下，**~80% 是 Qwen3-8B 在 DEV 上的真实 ceiling**。继续 prompt engineering 是负 ROI。

### 6.2 阶段 II 修正：DEV ≠ 真实 ceiling（§九/§十 实证）

读 FAQ 公布三任务权重 20/60/20 后，**重跑失败 6/2 在 OOD mock 上**的结果完全推翻上述结论：

| 失败 | 原始 DEV | OOD mock 重跑 | MCQ mock 重跑 | 新结论 |
|---|---|---|---|---|
| 失败 6（label 描述） | -1.7% | **+4.4%** | +0（auto skip） | 在 OOD 上**完全反转**，加权 +3.1% |
| 失败 2（reasoning） | -2.1% | **-1.5%** | **+3.4%** | 强任务异质性，自适应路由加权 +0.6% |

**修正后理解**：失败 1-8 的共同根因**不是"加东西没用"，而是"在 DEV 这个 in-distribution 场景下，加的信息与 few-shot 冗余"**。

- **DEV 上**：few-shot 已经给出强 in-context 信号，任何附加信息（reasoning/desc/shortlist）都是冗余稀释
- **OOD 上**：few-shot 因 lexical mismatch 失效，label 描述成为**唯一可用的语义信号**，收益巨大
- **MCQ 上**：任务需要推理链路（读题 → 对比选项 → 决策），reasoning 填补"解题心智模型"空缺

### 6.3 阶段 III 修正：Ceiling 是配置敏感的，不存在"全局最优"

§十 MCQ system_msg 又把 MCQ 推到 87%+，再验证同一 Qwen3-8B 在不同任务+不同 prompt 结构下达到的 ceiling 差异巨大：

| 任务 | 原始 ceiling（§六 阶段 I） | 当前 ceiling（§十二 后） |
|---|---|---|
| DEV | ~80% | **85.2%** |
| OOD | ~80% | **84.7%** |
| MCQ | ~81% | **87.5%** |

**最终结论（三次修正后）**：
1. **Ceiling 是配置敏感的**——不同任务在不同增强组合下的上限差异巨大
2. **没有单一"全局最优"配置**——任务自适应（基于 MCQ 检测等信号做路由）才是现实
3. **失败实验的价值随 base 变化**——早期 base 下的"失败"在更强 base 下可能反转，或只在特定任务子集上有效

**方法论教训**：每个失败实验都应记录"若 X 改变，结论可能反转"的条件。这是后续 §九/§十 两次反转的直接方法论动因。

## 七、跨部署实证（方法论修正）

### 7.1 问题

在 DashScope 上观察到"同 prompt 多次输出严格一致"（投票 N=3 token 完全等比）。最初归因为"服务端 cache / 低温采样"。但任务清单 §一 §6 写"4 次采样取均值"，暗示评测方部署是**真采样**——意味着投票在评测方下可能反而有效。

### 7.2 实验

切到 SiliconFlow（独立 API 服务，相同 `Qwen/Qwen3-8B`）做对照实验，用 [test/api_determinism_probe.py](test/api_determinism_probe.py) 探测：

| 测试 | DashScope | SiliconFlow |
|---|---|---|
| 同分类 prompt × 5 次输出 | 5/5 完全一致 | 5/5 完全一致 |
| 创意写作 × 3 次输出 | （未测） | **3/3 完全不同**（确证真采样） |
| 模糊分类 × 5 次（temp=1.0 / 1.5） | — | 5/5 一致 |
| 不同 `seed` × 5 次（同分类 prompt） | — | 5/5 一致 |
| **N=1 P1 baseline @ 539 dev** | **81.1%** | **79.4%** |
| **N=3 投票 @ 539 dev** | 81.1% | **79.8%（+0.4%）** |

### 7.3 两个独立结论

1. **失败 1 的归因要修正**：投票无效**不是**因为服务端 cache，**而是模型本身 logits peaked**——评测方部署也救不了投票。
2. **跨部署有 ±1-2% 浮动**：同模型同 prompt，DashScope vs SiliconFlow 差 1.7%（10 条/539）。差异来自 fp 精度（fp16 vs bf16）、kernel 版本、sampling 实现等不可控因素。**评测方分数不能从本地数字外推**——80~83% 都在合理范围。

### 7.4 方法论意义

以后做"对模型行为下结论"的实验前，**先用 [api_determinism_probe.py](test/api_determinism_probe.py) 探一下当前服务的行为基线**——能避免把"服务特性"当成"模型特性"或反之。

## 八、阶段 II 突破 A：RRF 检索融合（cos + BM25）

### 8.1 动机

P1 baseline 只用 IDF 加权 cosine。BM25 在信息检索文献里以"文档长度归一化"和"分布偏移鲁棒性"著称，且仅用 `collections.Counter` + 已维护的倒排索引即可实现，无额外开销。**三任务权重 20/60/20 公布后，OOD（60%）是主战场**，检索层的分布鲁棒性直接影响加权分。

### 8.2 实现

在已维护的倒排索引之上加 BM25 计算路径（[solution.py:119-144](solution.py#L119-L144)），与 IDF-cos 并存。**RRF 融合**（Reciprocal Rank Fusion，k=60）把两路 top-M 排名按 `1 / (k + rank)` 加权：

```python
for rank, (_, d) in enumerate(tfidf_ranked): rrf[d] += 1/(60 + rank + 1)
for rank, (_, d) in enumerate(bm25_ranked):  rrf[d] += 1/(60 + rank + 1)
```

### 8.3 三任务 × 三检索 完整消融

| 检索方法 | DEV (20%) | OOD (60%) | MCQ (20%) | 加权分 |
|---|---|---|---|---|
| IDF-cosine (P1) | 81.1% | 78.8% | 78.1% | 79.1% |
| BM25-only | 81.4% | 79.2% | 80.7% | 79.9% |
| **RRF 融合** | **81.4%** | **80.3%** | **81.2%** | **🏆 80.7%** |

### 8.4 关键结论

1. **RRF 三任务全面占优**——加权 +1.6% vs cos / +0.8% vs BM25
2. **DEV 上 BM25 ≈ RRF 是假象**——DEV 文本短、同分布，是 best case；**OOD 和 MCQ 才是分化场景**
3. **IR 文献中 RRF 的"分布偏移鲁棒性"被 OOD 实证**——RRF 比 BM25 多 +1.1%，比 cos 多 +1.5%
4. **cos 在长文本上显著差**——MCQ 题干最长 2694 char，cos 的 `|q|·|d|` 归一化被 char-gram 维度吹爆，BM25 的 `b=0.75` 文档长度归一化恰好修正这点（MCQ 上 BM25 比 cos 多 +2.6%）
5. **消融关键发现**：若只保留 BM25 去掉 RRF，DEV 仍 81.4%，但 OOD 掉 1.1% / MCQ 掉 0.5%——**RRF 的增量收益主要在分布偏移场景**

## 九、阶段 II 突破 B：LLM 生成 label 描述（失败 6 在 OOD 完全反转）

### 9.1 背景：失败 6 原始结论

2026-05-07 在 DEV 单任务上 lazy 生成 77 个 label 的描述塞入 prompt：**DEV -1.7%**（§5.6）。当时归因：
1. Prompt 长度 713 → 1410 翻倍，稀释 few-shot 信号
2. 描述是 LLM 自生成，风格趋同质量参差
3. 吸光偏置在解码层而非信息层

原结论：**"Label → 原型"思路在 8B 规模失效**。

### 9.2 重跑：FAQ 权重公布后的重新审视

根据 §六.2 方法论修正（"失败不是终点，是假设"），在 OOD mock（52 类跨域）上重跑失败 6：

### 9.3 变更内容

- `_ensure_descriptions()`：首次 predict 时按 10 labels/batch 调 LLM（OOD ~6 批，~15s）
- `_should_use_descriptions()`：若单字符 label ≥50% 判 MCQ，**自动 skip**（P0 后改为 text-stem 检测，见 §十二）
- `_build_messages` 内 label section 从 `LABEL_SET: a, b, c` 改为多行 `label: meaning`
- 每个 label 用 3 条原始样本作 prompt 上下文，要求生成 <20 词的 "discriminative not tautological" 描述
- 批量 prompt 用 `LABEL: name :: desc` 双冒号作解析锚点

### 9.4 三任务实证对比

| 任务 | 之前 (RRF base) | **+desc (P0)** | Δ | prompt 变化 |
|---|---|---|---|---|
| DEV (20%) | 81.4% | **83.7%** | **+2.3%** | 712 → 1663 |
| OOD (60%) | 80.3% | **84.7%** | **+4.4%** | 553 → 1097 |
| MCQ (20%) | 81.2% | 81.2% | 0 | 1400 持平（auto skip） |
| **加权** | 80.7% | **🏆 83.8%** | **+3.1%** | — |

### 9.5 为什么这次赢了？三重归因

> 同一机制在 DEV 上 -1.7% 翻成 +2.3%，在 OOD 上直接 +4.4%——反转幅度最大 6.1%。

**原因 1（prompt 结构）**：
- 失败 6：`label: description` 简单粘贴进 `LABEL_SET` 单行
- 现在：多行格式 `  label: meaning`，每 label 独占一行，`LABEL: name :: desc` 双冒号作解析锚点
- → LLM 更易把"描述"和"label 名"在注意力上解离

**原因 2（描述质量）**：
- 失败 6：prompt 没强调辨别性
- 现在：明确要求 "discriminative not tautological"、"under 20 words"、"based on example utterances"
- → 生成更锐利，语义边界更清晰

**原因 3（最关键：base 升级）**：
- 失败 6：base = IDF-cos (81.1%)，few-shot 召回噪声大；desc 稀释 few-shot 后无信号兜底 → **净负**
- 现在：base = RRF (81.4%)，检索本身更准；desc 成为**锦上添花的语义信号**，和正确 few-shot 叠加 → **净正**
- → **"底层先做对，上层再加层"的经典案例**

### 9.6 报告金句

> **失败不是终点，是假设。每个失败实验都应记录"若 X 改变，结论可能反转"的条件。**

失败 6 原归因是"Qwen3-8B 规模限制"——但真正的限制是 base 的检索噪声 + prompt 结构拙劣。换了 base 换了格式，同一机制立刻 -1.7% → +2.3%。

## 十、阶段 II 突破 C：任务自适应 reasoning（失败 2 部分反转）

### 10.1 三任务单独测 reasoning：强烈任务异质性

统一开启 reasoning（模型先给 ≤25 词 reasoning 再 `Label: <xxx>`）对比关闭：

| 任务 | 权重 | 无 reasoning（§九 P0 base） | 开 reasoning | Δ |
|---|---|---|---|---|
| DEV | 20% | 83.7% | **84.8%** | **+1.1%** |
| OOD | 60% | 84.7% | 83.2% | **-1.5%** |
| MCQ | 20% | 81.2% | 84.6% | **+3.4%** |
| **加权（统一）** | — | 83.80% | 83.80% | **0%（完美抵消）** |

### 10.2 异质性归因

- **MCQ +3.4%**：选择题天然需推理链路（题干→选项→决策），Qwen3-8B 标准 CoT 受益场景
- **DEV +1.1%**：同分布细粒度分类，reasoning 帮模型对照 few-shot 做微调（`card_arrival` vs `card_delivery_estimate` 边界思考）
- **OOD -1.5%**：反直觉但可解释——OOD 下 **label 描述已提供强语义信号**，reasoning 让 Qwen3-8B 产生**与描述冲突的自我分析**，回到失败 2"小模型 CoT 反作用"的机制。**语义信号冗余比信号缺乏更危险**。

### 10.3 任务自适应路由（当时采纳）

reasoning 收益**和 description 使用状态互斥**：

```python
use_reasoning = not self._descriptions  # 无描述 = MCQ → 开
                                         # 有描述 = DEV/OOD → 关
```

零成本设计——`_descriptions` 的状态在 `_ensure_descriptions()` 里已自动分叉，reasoning 路由复用这个信号。

**当时最终成绩**：

| 任务 | 配置 | 结果 |
|---|---|---|
| DEV | desc 开、reasoning 关 | 83.9% |
| OOD | desc 开、reasoning 关 | 84.2% |
| MCQ | desc skip、reasoning 开 | **85.4%** |
| **加权** | 自适应 | **🏆 84.4%（+0.58% vs §九）** |

### 10.4 第四重归因：信号互斥性

§九 总结的"底层先做对，上层再加层"之外，失败 2 的反转验证了第四重原因：

> **两个增强 A 和 B 各自在 base 上都有收益，但 A + B 一起上不一定叠加，甚至可能相互抵消。**

- **DEV + OOD**（有 desc）：desc 提供强语义 → 叠 reasoning 在 OOD 反而稀释（-1.5%）
- **MCQ**（无 desc）：无语义信号 → reasoning 填补空白（+3.4%）

→ **增强不可无脑叠加**。工程上需要"**互斥路由**"设计。

### 10.5 MCQ 专用 system_msg（再 +2%）

原 system_msg 对所有任务都是 "You are a strict text classifier"。对 MCQ 来说**任务定义错位**——Qwen3-8B 被当成普通分类器，没进入"解题"心智模型。改为：

> "You are solving a multiple-choice question. The input inside <query> contains a question and its options; choose the correct option label from LABEL_SET. First give ONE short sentence of reasoning..., then output your final answer on a new line as `Label: <exact label>`."

**MCQ 从 85.4% → 87.0-87.8%**（多次跑稳定 +2%）。**一句任务框架的描述换 9 条翻正**。

**这次结构 refactor 把 2 分支 system_msg 拆成 3 分支**（MCQ 解题 / 分类+reasoning / 分类 direct），同时修复了另一个候选变体 `bak_best_dev`（use_reasoning=True 但走分类器 system_msg）的语义错位 bug。

### 10.6 元教训（报告金句）

- §九：**底层做对，上层能反转**
- §十.4：**增强不能无脑叠加，需任务路由**
- §十.5：**任务框架错位和信号缺位一样致命——给 LLM 正确的任务定义和给它正确的语义信号同等重要**

MCQ system_msg 的 +2% 不来自任何新信息，**只来自"告诉 LLM 它在干什么"的一句话**。

## 十一、阶段 II 失败实验：朋友版本的 stopword + label-token boost 迁移失败

### 11.1 对比背景

对比朋友版本（FriendV = 单 BM25 + stopword + label-token 弱监督 + cap=3，加权 78.3%）后，尝试吸收其两个设计亮点叠加到加权最优 base：
- **stopword 过滤**：`_featurize` 里剔除 32 个英文 stopword
- **label 名 token 加权**：对每个 label 名 tokenize 提取 word token；在 `_search_bm25_ranked` 里对 query ∩ label 名 token 的 doc 加 `0.5 × sum(idf)` bonus

### 11.2 结果

| 任务 | 权重 | weighted base | P0.3 组合 | Δ |
|---|---|---|---|---|
| DEV | 20% | 83.9% | 82.2% | -1.7% |
| OOD | 60% | 84.2% | 81.7% | **-2.5%** |
| MCQ | 20% | 85.4% | 84.9% | -0.5% |
| **加权** | — | 84.4% | **82.4%** | **-1.96%** |

### 11.3 归因：信号空间竞争

朋友 base 没有 LLM desc，**stopword + label-token 是他获取语义信号的主要手段**；我们 base 已有 desc，叠加冲突：

- **机制 1**：banking 场景里 `my/your` 区分用户自己的卡还是商户的卡，有语义信息。过滤掉 stopword 使近义 label 边界模糊 → DEV -1.7%
- **机制 2**：label-token boost 把所有含 "card" 的 label 同时拉上 top-24；cap=2 × 8 条 few-shot 被同质化（6 条都是"card 相关"）；desc 原本能区分 `card_arrival` vs `card_linking`，现在 few-shot 信号让 LLM 困惑 → OOD -2.5%

**OOD 退步最严重（-2.5%）恰好印证假说**——OOD 是 desc 收益最大的场景，叠加词法 boost 冲突最严重。

### 11.4 教训：增强组件应该在"信号空缺维度"上工作

> 设计良好的系统里，每个新增组件应该在一个"之前空缺"的信号维度上工作。如果新组件和已有组件争夺同一信号空间，它只会把原本清晰的信号抖乱。

- **desc（§九）**：填补"LLM 无法从 few-shot 推出语义边界"的空缺 → 收益大
- **adaptive reasoning（§十）**：填补"MCQ 需推理链路"的空缺 → 收益大
- **stopword**：和 desc 争语义空间 → 抵消
- **label-token boost**：和 desc 争语义空间 → 抵消

朋友的 trick 在朴素 base 上是对的，迁移到富 base 就从正确变错误。

### 11.5 朋友版可吸收点的最终结论

| 朋友版本设计点 | 是否吸收 | 理由 |
|---|---|---|
| Stopword 过滤 | ❌ | 和 desc 冲突，损失 banking 细粒度区分 |
| Label 名 token 加权 | ❌ | 和 desc 冲突，同质化 few-shot |
| cap=3 | ❌ | 失败 3 已证伪 cap=1 负向；cap=3 方向未扩大验证但预期同样负向 |
| ` ``` ` 包裹 query | ❌ | `<query>` XML 包裹抗注入更稳（mock 9/9 证） |
| System msg 简短 | ❌ | 失败 7 已证伪 |

## 十二、阶段 III 加固：FAQ 细读后的鲁棒性修复

### 12.1 动机：MCQ 检测启发式的脆弱性

细读 FAQ（599 行）发现两个被忽略的风险点：

| FAQ 行 | 内容 | 对原检测器（单字符 label ≥50%）的冲击 |
|---|---|---|
| 行 379 / 403 | **MCQ 选项可能是 `apple/banana/cat/dog` 或 `1/2/3/4`，不保证单字符** | MCQ 被误判成 non-MCQ → 走 desc 生成 + 普通分类 system_msg → **丢掉 MCQ +3.4% 的 reasoning 收益** |
| 行 597 | "OOD 中会出现 label 为 A,B,C,D 的情况吗" — 官方未否认 | OOD 子任务可能恰好用 A/B/C/D → 被误判成 MCQ → 跳过 desc 生成 + 套上选择题 system_msg → 直接吞 OOD 上的 +4.4% |

**风险量化**：OOD 60% 权重 × 一次误判可能 -4.4% = -2.6 加权点；MCQ 20% × 一次误判 -3.4% = -0.7 加权点。**任何一次误判就把 84.5% 打回 ~82%**。

### 12.2 P0 修复：`_detect_mcq_task()` 改用 text stem 扫描

**新检测器**（[solution.py:428-445](solution.py#L428-L445)）：

```python
pat = re.compile(r'(?:^|\n)[ \t]*[A-Za-z0-9一二三四五六甲乙丙丁](?:\.|:|\))[ \t]+\S')
# 训练 text 含 ≥2 个选项 stem（A./A:/A)/1./一./甲. 等）
# ≥50% train 样本命中 → MCQ（FAQ 行 559 保证单任务内选项格式稳定）
```

**为什么比"单字符 label"启发式更鲁棒**：
1. MCQ 用 apple/banana 多字符 label 时，**label 信号失效，但 text stem 仍在**——text 必须给出选项内容（FAQ 行 320）→ 正确识别
2. 非 MCQ (OOD) 偶然用 A/B/C/D label 时，**label 信号误报，但 text 无选项 stem**（OOD 是正常句子）→ 正确排除

**mock 三任务实测**：

```
train_dev    N=231 stem-hit=  0/231 (0.0%)  → detected MCQ=False  ✓
train_ood    N=186 stem-hit=  0/186 (0.0%)  → detected MCQ=False  ✓
train_mcq    N= 96 stem-hit= 96/96  (100%)  → detected MCQ=True   ✓
```

**0%/0%/100% 分离**，远离 50% 阈值，任一失败模式都被覆盖。

同步把 `_build_messages` 里 `mcq_mode` 的计算来源从 `not self._descriptions` 改为 `_detect_mcq_task()`，**解耦 MCQ 信号与 desc 生成状态**（顺便修掉"desc 生成失败 → 误判 MCQ"的次生 bug）。

### 12.3 P1 修复：LLM 异常兜底改用检索 top-1

**原逻辑**：`predict` 的 except 分支 → `_fallback_label()` 返回训练集最高频 label。

**问题**：hard case 分析（§四）显示检索 top-1 命中率 47.3%，而最高频 label 命中率约等于 `1/num_labels` ≈ 1-5%。评测方 QPS 150/s 下偶发 429 或超时是可能的，每次兜底失误都是 -0.1% DEV。

**修复**（[solution.py:176-186](solution.py#L176-L186)）：

```python
def _retrieval_fallback(self, text: str) -> str:
    if text:
        try:
            hits = self._search(text, k=1)
            if hits: return self.memory[hits[0]][1]
        except Exception: pass
    return self._fallback_label()
```

### 12.4 变体对比：三个候选版本 4-run 实测

| 变体 | DEV (20%) | OOD (60%) | MCQ (20%) | 加权(mock) | P0/P1 安全 |
|---|---|---|---|---|---|
| `bak_best_dev` 原版 (use_reasoning=True, 无 P0P1) | 84.9% | 82.6% | 87.4% | 84.02% | ❌ |
| `weighted + P0P1` (use_reasoning=mcq_mode) | 83.6% | 82.7% | 87.5% | 83.84% | ✅ |
| **最终提交 (use_reasoning=True + P0P1)** | **85.2%** | 82.0% | 87.2% | 83.68% | ✅ |

**三版本加权分极差 0.34%，已进入日内噪声带**（OOD 单轮方差 1.7%）。

### 12.5 最终决策：`use_reasoning = True`

**决策依据**：DEV 是官方测试集、最可信代理。mock OOD/MCQ 是自造数据**可能偏乐观**。

三条逻辑：
1. **DEV +1.6% 是实打实的置信度提升**（4-run 方差仅 0.3%，每次都 85%）
2. **Mock 上 -0.16% 加权是在数据偏乐观假设下的估计**，实际可能不存在
3. **P0 + P1 双重加固**对冲了"use_reasoning=True 在 OOD 上可能 -1.5%"的风险——如果私有 OOD 真的有 reasoning 惩罚，也只在 60% 权重的**一个任务子集**上生效；而 DEV 的 +1.6% 是确定性收益

**数值上的成本/收益**：
- 成本（假设 mock OOD 完全真实）：-0.96% 加权
- 收益（DEV 是官方测试集的权重 20% 放大）：+0.32% 加权
- **但 DEV 是唯一可信代理**，当 mock 与可信代理冲突时应信可信代理 → 净收益为正

## 十五、设计选型理由（最终）

### 15.1 为什么选这套组合？

最终 Harness = **RRF 检索 + LLM label 描述 + 任务自适应路由（MCQ 专用 system_msg）+ 全任务 reasoning + P0/P1 鲁棒性修复**。

逐条理由：

1. **RRF 检索**（§八）：在三任务加权上最优，加权 +1.6% vs 原 cos-only。OOD 分布偏移场景下 RRF 的文献鲁棒性被直接实证
2. **LLM label 描述**（§九）：DEV +2.3% / OOD +4.4%，反转了 §5.6 失败 6 的 DEV 结论
3. **任务自适应路由**（§十）：MCQ 专用 system_msg（+2%）+ MCQ skip desc 生成（节省 ~15s 首次延迟）
4. **全任务 reasoning**（§十二.5）：DEV 是官方测试集，`use_reasoning=True` 比 `mcq_mode` 路由版本多 +1.6% DEV。mock OOD/MCQ 可能偏乐观，不值得为 mock 指标牺牲 DEV 置信度
5. **P0 MCQ 检测**（§十二.2）：text stem 扫描把 MCQ 判别的 false positive/negative 风险降到 mock 0%/0%/100% 分离
6. **P1 检索 fallback**（§十二.3）：LLM 429/超时时兜底命中率从 ~5% → ~47%

### 15.2 被证伪的对立选项

| 对立选项 | 证伪来源 |
|---|---|
| 收紧 LABEL_SET 到 shortlist | §5.4 失败 4 + §5.5 失败 5：漏召无救济通道 |
| 自一致性投票 | §5.1 失败 1 + §七 跨部署实证：模型 logits peaked，真采样下 +0.4% 噪声水平 |
| 不加 reasoning（纯分类器 direct） | §十.1 实证：DEV -1.1% / MCQ -3.4%，对官方 DEV 是失分 |
| 单 BM25（去掉 RRF） | §八.3：OOD -1.1% / MCQ -0.5%，RRF 收益主要在分布偏移 |
| cos-only | §八.3：全任务皆输 |
| Stopword 过滤 + label-token boost | §十一：和 desc 争信号空间，-2% 加权 |
| System msg 极简化 | §5.7 失败 7：省 token 代价 -0.4% |
| few-shot n=4/6/10/12 | §三 曲线实证 n=8 是严格单峰 sweet spot |

### 15.3 未跑但可能值得的方向

- **Constrained decoding**：logits 层强制 LABEL_SET 内，但任务接口不允许
- **训练样本 augmentation**（update 阶段调 LLM 改写）：测试 II 在 DEV 上 +0% / 3× 耗时，未在 OOD 上重跑
- **Query expansion**（predict 阶段 LLM 改写 query 后多路检索）：LLM 调用 ×2，超时风险增加
- **难例挖掘**（update 用 LLM 自测记录易混样本对）：调用预算大，未尝试

## 十六、局限性 + 未来方向

### 16.1 已知局限

- **检索 hard miss（§四 5% DEV）不可救**：每 label 仅 3 条训练样本是天花板
- **吸光 label 偏置（§四 14% DEV）模型固有**：超出 prompt engineering 边界
- **DEV 85% 不可外推到评测方**：跨部署 ±1-2% 浮动（§七）
- **mock OOD/MCQ 是自造数据可能偏乐观**：§十二.5 决策依据之一
- **reasoning 的 OOD 影响不确定**：§十.1 mock 上 -1.5%，但私有集未知；靠 P0/P1 加固分摊风险
- **Label 集接近 200 时 prompt 可能爆预算**：当前 4 级降级能 cap 到 2048，但 label section 会被收缩到 "label only, no desc"，降级后的语义信号丢失

### 16.2 未跑但可能的方向

- **在 OOD 上重跑失败 4/5**：当前 base（RRF + desc + adaptive reasoning）下，失败 4（二段式候选 shortlist）和失败 5（LIKELY 高亮）可能也有任务异质性
- **中文 unigram/bigram 检索特征**：当前 `_WORD_RE` 在中文上会把整段汉字当一个 token，char-gram 兜底但召回可能弱；单字 unigram 可补
- **注入样本的显式剥离**：当前仅 system msg + parser 白名单双层；mock 9/9 但攻击向量不全
- **私有集不确定性应对**：造更难的 mock 持续验证

### 16.3 已证伪 / 不再考虑

| 方向 | 证伪位置 |
|---|---|
| 自一致性投票（单任务 base） | §5.1 + §七 |
| DEV 上加 reasoning（单任务视角） | §5.2 — 任务自适应后仍采纳 |
| Per-label cap=1 | §5.3 |
| 二段式 LLM 调用（DEV base） | §5.4 |
| 候选 label 高亮 / 软提示（DEV base） | §5.5 |
| Lazy label 描述（cos base + 单行格式） | §5.6 — RRF base + 多行格式后反转 |
| System msg 极简化 | §5.7 |
| 朋友版 stopword + label-token boost（富 base） | §十一 |
